## Overview

The load balancer supports prompt cache (KV cache) reuse to reduce compute costs and latency. When a prompt is already cached on a backend, the backend can reuse the previously computed key-value pairs, significantly speeding up response generation.

This feature is implemented in:
- **BackendSelector**: [`selectBackendWithCache()`](../../llm-balancer/backend-selector.js#L234) - Cache-aware backend selection
- **PromptCache**: [`PromptCache`](../../llm-balancer/backends/PromptCache.js) - Per-backend cache storage
- **Balancer**: [`processQueueWhenBackendAvailable()`](../../llm-balancer/balancer.js#L340) - Queue processing with cache awareness
- **Balancer**: [`triggerQueueProcessing()`](../../llm-balancer/balancer.js#L560) - External queue processing trigger after cache operations

See also: [PROMPT_CACHE.md#how-cache-matching-works](#how-cache-matching-works)

## How Cache Matching Works

### Cache Entry Structure

Each cached prompt entry includes:
- **Prompt fingerprint** - A hash of the prompt content
- **Model** - The exact model name that generated this cache
- **Similarity score** - Measured similarity to the new request
- **Timestamp** - When the entry was cached

### Cache Hit Requirements

For a cache hit to occur, **BOTH** of the following must be true:

1. **Prompt similarity** >= 80% (configurable via `promptCacheSimilarityThreshold`)
2. **Model match** - The cached entry must be for the **exact same model** as the request

```
Cache Hit = (Prompt Similarity >= 0.8) AND (Model Match)
```

### Why Model Matching is Required

**KV cache is architecture-specific.** The cached key-value pairs are generated by a specific model's neural network architecture. A cached prompt for one model cannot be reused for a different model because:

1. **Different tokenization schemes** - Each model tokenizes text differently
2. **Different internal representations** - Models produce different embedding spaces
3. **Different layer structures** - Models have different architectures (number of layers, attention heads, etc.)

**Example:**
```
Backend 1 has cached: "write a story" for model "llama3"

Request 1: model="llama3", prompt="write a story"
  Result: ✅ CACHE HIT - Can reuse cached KV pairs

Request 2: model="qwen2.5", prompt="write a story"
  Result: ❌ NO CACHE HIT - Different model architecture
          Backend must compute new KV pairs from scratch
```

See tests: [backend-selector-cache-model-mismatch.test.js](../../llm-balancer/tests/unit/backend-selector-cache-model-mismatch.test.js)

## Backend Selection with Cache

### Cache-Aware Selection Algorithm

The `BackendSelector.selectBackendWithCache()` method uses the following logic:

1. **Check model support** - Verify any backend supports the requested model
2. **Check cache matches** - Search for cached prompts matching both content AND model
3. **Prefer cache hits** - If cache matches found, prefer that backend even if busy (for queuing)
4. **Fallback to priority** - If no cache matches, select by priority/availability

Implementation: [backend-selector.js#L234-L352](../../llm-balancer/backend-selector.js#L234-L352)

### Queue Behavior for Cache-Hit Backends

When a backend with a cache hit is at max concurrency:

```
Request arrives for model="llama3", prompt="write a story"
Backend 1: Has cache hit, but at maxConcurrency (activeRequestCount = 1)
Backend 2: No cache, but available

Decision: Return Backend 1 with status='busy'
          Balancer queues for Backend 1 specifically
          When Backend 1 becomes available, request uses cached KV pairs
```

This behavior achieves **~90%+ cache hit rates** instead of distributing requests across all backends.

See implementation: [balancer.js#L425-L428](../../llm-balancer/balancer.js#L425-L428)

## Limitations

1. **Single backend scope** - Cache is stored per backend, not shared across backends
2. **Model specificity** - Cache entries are model-specific and cannot be reused for different models
3. **Prompt variations** - Similar but not identical prompts may not hit cache if similarity < threshold
4. **Memory usage** - Each cached prompt consumes backend memory; evictions occur when cache is full

## Best Practices

1. **Batch similar prompts** - Group requests with similar prompts to maximize cache hits
2. **Use consistent model** - Route the same model to the same backend when possible
3. **Monitor cache hit rates** - Use `/debug/stats` to track effectiveness
4. **Adjust threshold carefully** - Lower threshold increases cache hits but may reduce accuracy
